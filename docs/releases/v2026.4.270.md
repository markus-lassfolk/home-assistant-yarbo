## Breaking changes — **Community Yarbo** rename

**Why:** The old domain `yarbo` collides with the official integration from [YarboInc/YarboHA](https://github.com/YarboInc/YarboHA). This project now uses **`community_yarbo`** everywhere that matters.

**What you need to do:**

1. **Remove** the old **Yarbo** (community) config entry from Home Assistant (Settings → Devices & services).
2. **Delete** the old folder `custom_components/yarbo/` from your config (or your HACS download path).
3. **Install** the new folder **`custom_components/community_yarbo/`** (HACS: reinstall or update the repository so you get the new structure).
4. **Restart** Home Assistant.
5. **Add** the integration again — it appears as **Community Yarbo**.
6. **Update anything that referenced the old domain:**
   - **Automations, scripts, and dashboards** — entity IDs will change prefix from `yarbo.` to `community_yarbo.` (or similar, depending on how HA renamed entities).
   - **Services** — use `community_yarbo.*` service names instead of `yarbo.*`.
   - **Event bus** — events are now named like `community_yarbo_job_completed` instead of `yarbo_job_completed`.
   - **Blueprints** — use the blueprint path under `blueprints/automation/community_yarbo/` and selectors that target `integration: community_yarbo`.
7. **MQTT debug recordings** (if you use them) now live under **`community_yarbo_recordings/`** instead of the old path.

There is **no automatic in-place migration** of the old config entry; a clean remove → install → re-add is the supported path.

## Reliability and bug fixes

### Clean shutdown and background tasks

Background telemetry work could hit a **closed event loop** during shutdown (for example when MQTT or thread callbacks fired late). The coordinator now handles that case and uses a small helper for retry sleeps so shutdown is less noisy and less likely to log tracebacks. (Crash telemetry ref. GlitchTip #137; follow-up in PR [#154](https://github.com/markus-lassfolk/home-assistant-yarbo/pull/154).)

### Controller timeouts you can understand

If talking to the robot **times out** (busy broker, robot asleep, library timeout), you now get a **clear Home Assistant error** instead of a raw failure path. Buttons, switches, lights, numbers, select, lawn mower, the coordinator, **services**, and **repairs** all go through a shared path that maps library timeouts to user-visible messages, without incorrectly triggering “controller lost” repairs for a simple timeout. ([#147](https://github.com/markus-lassfolk/home-assistant-yarbo/issues/147))

### Broker configuration edge cases

Some older or partially migrated entries could be missing **`broker_host`**, which led to **KeyError** crashes. The integration now **resolves** the host from `broker_endpoints` or alternate fields, **normalizes** entry data where possible, **migrates** legacy data, and fails setup with a clear **`ConfigEntryError`** if the broker configuration is unusable — instead of failing deep inside the coordinator. ([#155](https://github.com/markus-lassfolk/home-assistant-yarbo/issues/155))

### Options flow and unload

Changing integration **options** or **unloading** the integration is hardened against **stale or partial runtime data** (for example if internal state was already torn down). This avoids **KeyError** surprises when options propagate. ([#148](https://github.com/markus-lassfolk/home-assistant-yarbo/issues/148))

### Low-battery event logic

When the robot goes **offline**, battery capacity can be **unknown**. The low-battery listener no longer compares `None` to integers, which could crash a background thread. ([#146](https://github.com/markus-lassfolk/home-assistant-yarbo/issues/146))

### Library version vs manifest (get_controller timeout)

The manifest requires **`python-yarbo>=2026.3.21`**. The integration’s minimum check is aligned so an **older library** cannot slip past and then break when **`get_controller(timeout=...)`** is used (which caused a **TypeError** in the wild). ([#153](https://github.com/markus-lassfolk/home-assistant-yarbo/issues/153), GlitchTip #30.)

### Error reporting and Sentry

- CI and tests include a proper **`yarbo.error_reporting`** stub so imports match production behaviour.
- The integration again calls the **library’s** `init_error_reporting(enabled=False)` where appropriate so **python-yarbo** does not auto-initialize Sentry in parallel with the integration — avoiding **duplicate** crash reports.

## Internationalization

**Nine new translation files** ship with this release:

- **Finnish (fi)** — community contribution, cleaned up for consistency  
- **Swedish (sv), German (de), Dutch (nl), Norwegian Bokmål (nb), Spanish (es), French (fr), Italian (it), Polish (pl)**

All files are validated to match **English keys** (no missing or stray strings). Copy was tuned for **informal “you”** where that fits the language (e.g. German *du*, Italian *tu*, French *tu*), **sentence case** UI labels, and translated product terms (Smart Vision, Cover, Blowing, etc.).

## Developer experience, tests, and CI

- **Ruff** — code and tests are formatted and lint-clean (import order, line length, annotations, `assert False` → `raise AssertionError`, etc.).
- **`CONFIG_SCHEMA`** — empty config schema registered for the domain for broader Home Assistant compatibility.
- **DHCP discovery tests** — `async_discover_endpoints` is mocked where needed so **pytest-socket** does not block the suite on real sockets.
- **CI** — `python-yarbo` is checked out from **`main`** (the library repo has no `develop` branch), fixing broken installs in pipelines.
- **GitHub Actions** — routine bumps (e.g. `actions/github-script` v9, `softprops/action-gh-release` v3).
- **setuptools** (dev) — lower bound raised for modern build tooling ([#151](https://github.com/markus-lassfolk/home-assistant-yarbo/issues/151)).
